Timepoint Pro provides built-in evaluation metrics to validate simulation quality and consistency.

## Running Evaluation

```bash
python cli.py mode=evaluate
```

Evaluates all entities currently stored in the database.

## Temporal Coherence Score

Measures behavioral consistency across timepoints.

### Formula

```python
violations = 0
for each consecutive timepoint pair:
    if personality_traits_changed_significantly:
        violations += 1

score = 1.0 - (violations / num_transitions)
```

### What It Validates

<AccordionGroup>
  <Accordion title="Behavioral Inertia" icon="weight-hanging">
    Personality traits should remain stable over time
    
    **Checks:**
    - Personality trait consistency
    - Character arc plausibility
    - No sudden personality shifts
    
    **Example violations:**
    - Cautious character becomes reckless overnight
    - Reserved person suddenly becomes extroverted
    - Core values change without cause
  </Accordion>
  
  <Accordion title="Trait Persistence" icon="fingerprint">
    Core characteristics persist unless causally justified
    
    **Checks:**
    - Trait stability across timepoints
    - Gradual vs. sudden changes
    - Causal explanations for shifts
  </Accordion>
</AccordionGroup>

### Score Interpretation

<Tabs>
  <Tab title="1.00">
    **Perfect Coherence**
    
    No behavioral violations detected. Entities maintain consistent personalities across all timepoints.
  </Tab>
  
  <Tab title="0.80-0.99">
    **High Coherence**
    
    Minor violations, likely justified by causal events. Good temporal consistency.
  </Tab>
  
  <Tab title="0.60-0.79">
    **Moderate Coherence**
    
    Some behavioral inconsistencies. May need review of temporal transitions.
  </Tab>
  
  <Tab title="<0.60">
    **Low Coherence**
    
    Significant violations. Entities showing unstable or inconsistent behavior.
  </Tab>
</Tabs>

## Knowledge Consistency Score

Validates information conservation - entities can only know what they've been exposed to.

### Formula

```python
if knowledge_properly_sourced:
    score = 1.0
else:
    score = 0.0
```

### What It Validates

<AccordionGroup>
  <Accordion title="Exposure Event Tracking" icon="eye">
    Every knowledge item must have a source
    
    **Checks:**
    - All knowledge has recorded exposure event
    - Source entity or event exists
    - Timestamp is causally valid
    
    **Example violations:**
    - Entity knows information without witnessing it
    - Knowledge appears without source
    - Anachronistic information (knows future events)
  </Accordion>
  
  <Accordion title="Information Propagation" icon="share-nodes">
    Knowledge spreads through valid paths
    
    **Checks:**
    - Information flows along relationship edges
    - No spontaneous knowledge generation
    - Social network constraints respected
    
    **Example violations:**
    - Entity knows secrets without connection to source
    - Information spreads faster than possible
    - Knowledge crosses disconnected graph components
  </Accordion>
  
  <Accordion title="Temporal Causality" icon="arrow-right">
    Knowledge can only come from past events
    
    **Checks:**
    - Exposure timestamp < current timepoint
    - No future information leak
    - Proper causal chain
    
    **Example violations:**
    - Entity knows outcome before it happens
    - Future information influences past decisions
    - Causal chain broken
  </Accordion>
</AccordionGroup>

### Score Interpretation

<Tabs>
  <Tab title="1.0">
    **Valid**
    
    All knowledge properly sourced. No information conservation violations.
  </Tab>
  
  <Tab title="0.0">
    **Invalid**
    
    Knowledge consistency violation detected. Entity has information without valid exposure.
  </Tab>
</Tabs>

## Biological Plausibility Score

Measures constraint enforcement and physical/resource realism.

### Formula

```python
violations = 0
for each action:
    if violates_constraints:
        violations += 1

score = 1.0 - (violations / num_actions)
```

### What It Validates

<AccordionGroup>
  <Accordion title="Physical Constraints" icon="ruler">
    Actions respect physical limitations
    
    **Checks:**
    - Movement speed plausible
    - Energy expenditure realistic
    - Physical capabilities within human range
    
    **Example violations:**
    - Entity travels impossible distance in timespan
    - Action requires more energy than available
    - Superhuman abilities without justification
  </Accordion>
  
  <Accordion title="Resource Constraints" icon="gas-pump">
    Actions consume appropriate resources
    
    **Checks:**
    - Energy budget tracking
    - Resource availability
    - Consumption rates
    
    **Example violations:**
    - Entity acts without sufficient energy
    - Resource consumption exceeds supply
    - Negative resource balances
  </Accordion>
  
  <Accordion title="Embodied States" icon="person">
    Physical and emotional states influence behavior
    
    **Checks:**
    - Fatigue affects performance
    - Stress influences decisions
    - Physiological needs matter
    
    **Example violations:**
    - Exhausted entity performs at peak
    - Emotional state ignored in decision-making
    - Physical needs not reflected in behavior
  </Accordion>
</AccordionGroup>

### Score Interpretation

<Tabs>
  <Tab title="1.00">
    **Fully Plausible**
    
    No constraint violations. All actions respect physical and resource limitations.
  </Tab>
  
  <Tab title="0.80-0.99">
    **Highly Plausible**
    
    Minor violations, likely edge cases. Generally realistic.
  </Tab>
  
  <Tab title="0.60-0.79">
    **Moderately Plausible**
    
    Some unrealistic actions. May need constraint tuning.
  </Tab>
  
  <Tab title="<0.60">
    **Low Plausibility**
    
    Many violations. Entities behaving unrealistically.
  </Tab>
</Tabs>

<Tabs>

  </Tab>
  
  <Tab title="SCENE">

    ~500-1000 tokens per entity
  </Tab>
  
  <Tab title="DIALOG">

  </Tab>

## Validation Integration

Evaluation metrics use the same validators as training:

- `validate_behavioral_inertia()` - Temporal coherence
- `validate_information_conservation()` - Knowledge consistency
- `validate_biological_constraints()` - Biological plausibility

See [Validation](../core-concepts/validation) for implementation details.

## When to Evaluate

Run evaluation after:

<Steps>
  <Step title="Training">
    After `mode=train` or `mode=temporal_train` to validate entity quality
  </Step>
  
  <Step title="Simulation">
    After running templates with `./run.sh` to check consistency
  </Step>
  
  <Step title="Debugging">
    When investigating unexpected entity behavior
  </Step>
  
  <Step title="Before Export">
    Before exporting data to ensure quality
  </Step>
</Steps>
